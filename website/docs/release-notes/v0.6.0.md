# Gitmoot v0.6.0

Makes same-repo parallel jobs **discoverable**: requesting multiple daemon
workers now actually delivers parallelism, the scheduler/worker config is visible
in `daemon status`, and a serializing config that meets a parallel workload now
warns with the exact relaunch command instead of silently running one-at-a-time.

## What's new since v0.5.2

### Daemon — parallel jobs on a repo (#444)

- **feat(daemon): `--workers > 1` auto-selects `--scheduler pool`.** Requesting
  multiple workers under the default per-tick `barrier` serialized same-repo jobs
  anyway (one checkout lock), so it was almost never what anyone wanted. When more
  than one worker is requested and `--scheduler` is left unset, the daemon now
  selects `pool`. **This is a behavior change** vs the v0.5.0 "default unchanged"
  promise — but only when you asked for multiple workers. The single-worker default
  is unchanged, and an explicit `--scheduler barrier` always forces the old
  per-tick behavior.
- **feat(daemon): `--parallel N` convenience flag.** Sets `--workers N` and
  `--scheduler pool` together, named for the goal rather than the mechanism. It
  cannot be combined with `--workers` or `--scheduler`.
- **feat(daemon): `daemon status` reports scheduler + workers.** A new
  `scheduler: <barrier|pool>, workers: <n>` line answers "is the daemon configured
  for the parallelism I asked for?" without reading launch flags; it flags a
  multi-worker `barrier` config and points at the fix.
- **feat(daemon): preflight serialization warning.** When ≥2 parallelizable jobs
  (same repo, dependency-unblocked, **distinct runtime sessions**) are queued under
  a serializing config, the daemon logs the exact relaunch command instead of
  silently serializing them.
- **docs: "Run jobs in parallel on a repo" guide** — covers the on-ramp and the
  **two** serialization layers (checkout lock and the `runtime:<runtime>:<ref>`
  session lock), so the docs aren't misleading about what actually parallelizes.

### Deferred follow-ups

Scoped out of this release deliberately, to keep it reviewable:

- **Top-level `implement` auto-isolation.** `pool` still isolates only read-only
  `ask`/`review` jobs; a plain same-repo `implement` job with no worktree continues
  to serialize. Parallelizing those via the daemon needs implement-eligible
  auto-isolation (branch worktree + branch-lock handling + a worktree cap and
  disposal sweep).
- **`daemon reconfigure`.** A drain-not-drop reconfigure that preserves daemon auth
  so changing workers/scheduler doesn't require a full restart.

## Upgrade note

If you run the daemon with `--workers > 1` and rely on `barrier` semantics, add an
explicit `--scheduler barrier` to keep the old behavior. Everyone else gets the
parallelism the worker count implies.

## Install / upgrade

- Existing install: `gitmoot update`
- Fresh install: `curl -fsSL https://gitmoot.io/install.sh | sh`
